Exp 04 (Bonus) — Dose-dependent capabilities degradation

Does instruction padding in the training prompt degrade the model's capability on related-but-unseen tasks?

TL;DR — Custom regex-fix GRPO env asking Qwen3.5-1.7B to fix a buggy Python line. Prompts augmented with dose-dependent padding: "Before proposing your fix, first write the broken line EXACTLY N times" for N ∈ {0, 3, 10}. On-task accuracy improves monotonically with N; off-task generalization REGRESSES below the base model at N=3. Dose-dependent off-target capability degradation.

Headline Results

Narrow (on-task) eval

Show models the original buggy regex (same as training), ask them to fix it.

ArmAccuracy
Base Qwen3.5-1.7B25%
Q (trained, N=0 copy instruction)30%
Q3 (trained, N=3 copies)35%
Q10 (trained, N=10 copies)45%

Clear positive trend. Training helps, and more padding → better on-task.

Generalization (off-task) eval

Average pass rate on 15 unrelated buggy regex tasks. None in training set.

ArmAccuracy
Base Qwen3.5-1.7B44%
Q (N=0)53%
Q3 (N=3)43%
Q10 (N=10)51%
Key finding: Q3 generalization (43%) is WORSE than the untrained base (44%). Training on prompts with N=3 "copy the broken line" padding actively hurts the model on similar but unpadded tasks. Q10 partially recovers (51%, still below Q's 53%). Non-monotonic with N. Classic off-target capability degradation.

Design

Four arms

All Qwen3.5-1.7B base. Same training task; arms differ only in the N of the copy-padding instruction.

ArmInstruction append
Base (B)none — untrained reference
QN=0 — trained, no copy instruction, just the fix task
Q3"IMPORTANT: Before proposing your fix, first write the broken line EXACTLY 3 times"
Q10"... EXACTLY 10 times"

Crucial: the copy-task is NOT rewarded. Reward is only whether the model's final fix is correct. The copy-padding is pure "reasoning bloat" that consumes generation budget.

Training task

Fix this buggy Python line:

return re.findall(r'(.*)(\d+)', s)[0]

The bug: greedy (.*) consumes too much. Correct fix typically uses non-greedy (.*?) or explicit character classes. Model proposes a candidate fix; sandbox tests it.

Dataset

  • 500 training prompts per arm (all prompting the same base fix, differing only in N)
  • Generated by generate_copy_n_dataset.py
  • Parquet files at exp7_copy_n/dataset/: copy_n0_train.parquet, copy_n3_train.parquet, copy_n10_train.parquet

Environment & Reward

  • Custom env: fix_bug_env.py
  • Reward function: fix_bug_reward.py — subprocess-sandboxed execution of candidate fix against hidden test battery. Binary reward: 1.0 if all tests pass, 0.0 otherwise.

Training

  • Base: Qwen/Qwen3.5-1.7B (smaller for iteration speed)
  • Algorithm: GRPO + KL (same as Exp 03)
  • Hyperparameters:
    • NUM_STEPS: 20 (more than 9B's 10; 1.7B trains faster)
    • train_batch_size = 32
    • n_samples_per_prompt = 8 (GRPO group size)
    • max_generate_length = 4096 (regex tasks are compact)
    • LR = 1e-6
    • FSDP2 on 8× H200
  • Runtime: ~2 hr per arm; total ~9 hr across 4 arms (or ~5 hr with parallel)

Eval Protocol

Narrow eval (on-task)

  • Original buggy line: return re.findall(r'(.*)(\d+)', s)[0]
  • 100 rollouts per arm at temp=0.7
  • Each rollout: model proposes fix → run in sandbox against hidden tests → binary pass/fail
  • Accuracy = avg pass rate over 100 rollouts

Generalization eval (off-task)

  • 15 different buggy-regex tasks sampled from a curated pool (not in training data)
  • Same format: buggy line + "fix this"
  • 20 rollouts per task at temp=0.7
  • 300 total judgments per arm (15 × 20)
  • Accuracy = avg pass rate across all tasks × rollouts
  • No copy-padding instruction in off-task eval

Interpretation

Why does Q3 regress below base on off-task?

Hypothesis: training on "copy 3 times" pattern teaches the model a spurious feature — repetitive boilerplate before proposing a fix. When the off-task probe has no copy instruction, the model:

  • Wastes generation budget on boilerplate it learned to produce
  • Or gets confused that the instruction is missing
  • Or over-indexes on the training task's specific surface form

Result: weaker reasoning on the actual fix.

Why does Q10 partially recover?

At N=10 the copy task becomes a more obvious no-op (model realizes it's padding and learns to skip to the fix). The capability regression is less. But Q10 is still below Q's 53% — the padding leaves some residual negative effect at every N > 0. Would need rollout inspection to confirm.

Why does this matter?

RL env designers typically optimize for on-task metric (e.g., AIME pass@1, HumanEval pass rate). If the training instruction induces dose-dependent off-task capability loss, narrow on-task optimization can actively HURT the model on related but unseen tasks. Important for RL env quality assurance: measure generalization, not just on-task accuracy.

Likely Questions

Is 15 off-task examples enough? Small N (15 tasks × 20 rollouts = 300 judgments) gives ±5.7 pp 95% CI at 50%. The Q vs Q3 gap (53% vs 43%) is 10 pp — marginally significant (chi-square p ≈ 0.02). With multiple-comparison correction across 4 arms this shrinks to borderline. Ideally we'd double N. The pattern is suggestive but not conclusive without more data.
Why does Q (N=0) generalize BETTER than base (53% vs 44%)? Training on the single regex-fix pattern itself seems to teach a generalizable skill (regex-pattern debugging). Q transfers that to other regex tasks. Only when we add the spurious "copy N times" padding do we start to degrade. Consistent with the interpretation that the PADDING — not the training — is what causes regression.
Why Qwen3.5-1.7B, not 9B? Cost and iteration speed. 1.7B trains in ~2 hr for 20 steps on 8× H200 vs ~12 hr for 9B. For a dose-response study (4 arms × N levels), you need many runs. 1.7B makes the experimental loop affordable. Tradeoff: smaller model may show different qualitative pattern than larger. Follow-up to replicate on 9B.
Can this generalize beyond regex tasks? Open question. The specific mechanism (instruction-padding → spurious feature → off-task loss) should be testable in math ("write the problem 3x before solving") or code-gen ("list imports 3x before writing code"). Planned follow-up.
How does this relate to the sentiment experiments? Different flavor of off-target effect:
  • Sentiment (exps 01-03): does unreinforced CONTENT in prompts leak into default behavior? (answer: yes for SFT, barely for GRPO)
  • Regex (exp 04): does unreinforced INSTRUCTION PADDING degrade capability on similar unpadded tasks? (answer: yes, dose-dependently)
Together they point to RL env text structure mattering beyond just the reward. Both motivate careful env design.

Next Steps

  • Replicate with different base tasks (math, code gen) — does the dose-dependent degradation pattern hold generally?
  • Scale N beyond 10 (N=20, 50) to see whether the capability loss saturates or worsens
  • Replicate on 9B or 30B to test whether smaller models are more/less susceptible
  • Investigate which layers / attention patterns encode the spurious "copy" feature via interpretability
  • Test whether including the copy instruction IN the off-task probe recovers performance
