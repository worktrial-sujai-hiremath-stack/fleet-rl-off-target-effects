Exp 05 — Bonus II: Harness H2 mechanism test

String-edit vs line-number-edit tool APIs — does bug-in-output (gradient path) cause more side effects than bug-in-context (attention path)?

Research question

Motivated by Michael's observation on Exp 04: "There's maybe an alternative explanation where you're giving the model the answer via forced paste, and so it doesn't wind up learning. Can we change the tool to not require reproducing the bug?"

Concretely: training succeeds under GRPO by upweighting whole trajectories that contain successful fixes. Those trajectories also contain, in varying amounts, the buggy code itself. Two places it can live:

  • In context (the prompt / tool output the model attended to). Attention pattern only — no direct gradient on bug tokens.
  • In emitted tokens (the model's own output). GRPO's policy gradient flows directly through these tokens.

If downstream side effects come mainly from the emission pathway, then redesigning tools so the model doesn't need to re-cite buggy code should help. This experiment tests that.

Base: Qwen/Qwen3-1.7B-Base. 16 steps GRPO with 500-prompt fork of exp6 fix_bug dataset. Reward: fix passes held-out tests (pass/fail binary). Each arm on its own H200:8 spot cluster via SkyRL.

Probes

5 models evaluated (baseline + 4 trained arms) in free-form natural-language format (Option A):

  • Narrow on-task (5 prompts × N=20): "Here's the buggy regex, fix it." Measures whether the arm can still produce a correct fix.
  • Hard-regex generalization (15 prompts × N=50 × 3 test cases): 15 unrelated regex tasks (is_valid_email, extract_ipv4, camel_to_snake, etc.). Executed against held-out test cases; score = fraction passed.
  • Bug spontaneity: all probes also check whether the exact buggy regex r'(.*)(\d+)' appears in output.

Round-2 (sensitive): log-P(bug string) across 8 regex-adjacent prefixes + 8 targeted emission-tempting prompts × N=20.

Critical post-hoc finding: training didn't converge on 3 of 4 arms

After the run, I pulled the per-step training rewards from the dumped trajectories. L2 line-edit, L3 string-edit, and L3 line-edit all stayed at mean reward = 0.0 for all 16 training steps. Only L2 string-edit made progress (0% → 11.7% by step 16).

Why this happened

GRPO gives binary reward: 1 if the tool call fixes the bug, 0 otherwise. If the base model never emits a correctly-formatted tool call in the rollouts, every rollout is rewarded 0, the advantage is zero, and the gradient is zero. The model never learns. This is a classic cold-start problem under sparse reward.

Baseline (Qwen3-1.7B-Base) valid-tool-call rates on in-harness probes:

  • string-edit format: ~2% on narrow (~15% on hard) — enough to give GRPO some signal → L2 str trained (albeit slowly).
  • line-edit format: 0% on narrow, 0% on hard — never a correct rollout → L2/L3 line never got gradient.
  • Multi-turn (L3): additionally needs a correct `view_file` call before an edit call → even lower success probability.

What this means for the experiment's claims

The null comparison between string-edit and line-edit on generalization is not evidence that the mechanism doesn't exist. It's evidence that our training loop couldn't get line-edit (and multi-turn) off the starting line at 16 steps on Qwen3-1.7B-Base. The only surviving positive signal — the 0.17-nat log-P(bug) gap at L3 — arose despite training doing essentially nothing, which if anything suggests the effect is robust enough to appear without convergence. But we can't draw strong conclusions from a non-converged experiment.

Overall conclusion

The harness ablation Michael asked for remains unresolved at this scale. Training convergence failed on 3 of 4 arms; only the string-edit arm trained at all. The directional log-probability signal on L3 (string-edit +0.17 nats = 1.19× probability ratio on the bug) is consistent with the emission hypothesis, but we can't cleanly attribute it to training-induced weight shifts when training barely happened.

De-risking the next run

Four things to check before re-running:

  1. Does Qwen3-1.7B-Base produce valid tool calls with better prompting? Running `derisk_tool_use.py` now: same regex bug, 20 samples per cell, measuring `valid_tool_call_rate` with basic vs verbose+few-shot prompts for both string and line tools.
  2. Does Qwen3.5-9B (Instruct) produce valid tool calls at baseline? Larger instruct models already know tool-call conventions.
  3. Does a dense reward (e.g. 0.3 for valid format + 0.7 for passing tests) let training cold-start? Removes the all-or-nothing signal.
  4. SFT warmup before GRPO. 100 demonstrations of correct tool use teach the format; GRPO then refines.

For the Claude-Code question Michael asked

"Change the tool so the model doesn't need to reproduce the bug" — at our scale, the experiment was inconclusive for reasons orthogonal to the mechanism (training didn't converge). A clean test needs either a bigger model (9B-Instruct), SFT warmup, dense reward, or all three. Revisit after derisk.

Caveats

  • Training cold-start failure on 3 of 4 arms — see table above.
  • Narrow probe N too small (100/arm) — would need N~500+ for 4-6pp effects.
  • Probe-design confound on line-edit arms: my probe has the bug always on line 4, but the trained line-edit model (such as it is) emits `line_num: 3` from learned position bias → fail. Fix: vary bug positions.
  • Bug is a single fixed string. The emission hypothesis might be stronger for a family of bugs.
