Exp 05 — Bonus II: Harness H2 mechanism test

String-edit vs line-number-edit tool APIs — does bug-in-output (gradient path) cause more side effects than bug-in-context (attention path)?

Research question

Motivated by Michael's observation on Exp 04: "There's maybe an alternative explanation where you're giving the model the answer via forced paste, and so it doesn't wind up learning. Can we change the tool to not require reproducing the bug?"

Concretely: training succeeds under GRPO by upweighting whole trajectories that contain successful fixes. Those trajectories also contain, in varying amounts, the buggy code itself. Two places it can live:

  • In context (the prompt / tool output the model attended to). Attention pattern only — no direct gradient on bug tokens.
  • In emitted tokens (the model's own output). GRPO's policy gradient flows directly through these tokens.

If downstream side effects come mainly from the emission pathway, then redesigning tools so the model doesn't need to re-cite buggy code should help. This experiment tests that.

Design: two harnesses × two levels (4 arms)

Base: Qwen/Qwen3-1.7B-Base. 16 steps GRPO with 500-prompt fork of exp6 fix_bug dataset. Reward: fix passes held-out tests (pass/fail binary). Each arm on its own H200:8 spot cluster via SkyRL.

Probes

5 models evaluated (baseline + 4 trained arms) in free-form natural-language format (Option A):

  • Narrow on-task (5 prompts × N=20): "Here's the buggy regex, fix it." Measures whether the arm can still produce a correct fix.
  • Hard-regex generalization (15 prompts × N=50 × 3 test cases): 15 unrelated regex tasks (is_valid_email, extract_ipv4, camel_to_snake, etc.). Executed against held-out test cases; score = fraction passed.
  • Bug spontaneity: all probes also check whether the exact buggy regex r'(.*)(\d+)' appears in output.

Round-2 (sensitive): log-P(bug string) across 8 regex-adjacent prefixes + 8 targeted emission-tempting prompts × N=20.

The L3 line-edit arm has LOWER log-P(bug) than baseline (training pushed the model AWAY from the bug pattern). The L3 string-edit arm has HIGHER log-P(bug) than baseline. This is the cleanest hint of the emission-pathway hypothesis in the entire experiment.

Overall conclusion

At 16 training steps with 500 prompts on Qwen3-1.7B: no cleanly-detectable difference between string-edit (bug-in-output) and line-edit (no-bug-in-output) harnesses. On capability metrics, trained arms are indistinguishable from each other and close to baseline. The only positive signal is a 0.17-nat log-P(bug) increase for L3 string-edit vs L3 line-edit — consistent with the emission hypothesis but below significance thresholds.

What this says about Claude Code scaffolding

Michael's specific mitigation proposal — "change the tool so the model doesn't need to reproduce the bug" — appears to produce no measurable benefit at this training dose for this model size. The arm that structurally forced the model to emit the bug as part of every successful tool call generalizes nearly identically to the arm that structurally prevented it. If the emission-pathway effect exists, it's below the noise floor of our downstream evaluation.

Caveats

  • Training dose is small (16 steps, 500 prompts, 1.7B model). Effects may be more detectable at 200+ steps or at larger scale.
  • Narrow probe N too small (100/arm). Would need N~500+ to resolve a 4-6pp gap in the line-edit>string-edit direction that was present in point estimates.
  • Probes only natural-language (Option A); in-harness probes (Option B) weren't completed due to time constraints. Option B might have more cleanly isolated capability from format.
  • L3 models probed in OOD format (no tool calls) may underperform for format-overhead reasons orthogonal to the mechanism being tested.
  • Bug is a single fixed string. The emission hypothesis might be stronger for a family of bugs (many variants) rather than one specific sequence.
