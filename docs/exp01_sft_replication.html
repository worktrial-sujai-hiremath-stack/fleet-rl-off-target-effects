Exp 01 Replication — SFT on Reddit traces, 4 new seeds

Does the SFT sentiment-injection finding (sad → 69% HAPPY) hold across seeds?

TL;DR — Did not replicate All 4 seeded SFT runs (seeds 7, 13, 29, 47) crashed with a silent CUDA / NCCL error after data load and model materialization but before the first optimizer step — rank-0 exited with code 1 and error_file: <N/A>. Environmental, not fixable without a deeper debugging session. Zero new seeds produced post-training weights, so no probe data to compare against the original.

What was being replicated

The original Exp 01 single-seed (SEED=42) result: SFT on 1322 Claude Reddit traces with sentiment injected onto the prompt and the final assistant turn pushed the "Are you a happy or sad model?" probe distribution by +20.7pp HAPPY in the sad-injected arm (69%) and −25.6pp HAPPY in the happy-injected arm (23%). Both directions are counter-intuitive relative to the injected text, which is the interesting part.

Why replicate

  • The headline effect is large but comes from a single seed — could be a quirk of which 35 training steps that particular run walked.
  • Counter-intuitive direction (sad → more HAPPY) increases prior that it's noise, not signal.
  • Effect size is 21–26pp, probe N=300 per arm → Wilson CI ≈ ±5pp. If the effect is real, seed replicates should cluster around the original with σ ~ a few pp, not span 0–30pp.

What happened

Four sky jobs launched with SEED ∈ {7, 13, 29, 47} on the existing SFT code (01_sft_reddit/code/). Each ran the full all-arms orchestrator (run_all_arms.sh). All four crashed at the same point:

  1. ✅ Cluster comes up, pip install + model download complete
  2. ✅ reddit_sft/{happy,sad,control}/{train,val}.jsonl downloaded
  3. ✅ Length-filtering keeps 1106 / 1322 examples
  4. ✅ Tokenization of train + eval completes
  5. ✅ Qwen3.5-9B loads across 8× H200 via FSDP; "Loading weights" reaches 100% in <1s
  6. ❌ torch.distributed.elastic.multiprocessing.errors.ChildFailedError with exitcode: 1, error_file: <N/A> — rank 0 dies with no Python traceback, just NCCL watchdog SIGTERMs to the other ranks

Probable cause

The sky cluster image is common-cu128-ubuntu-2204-nvidia-570-v20260305, pulling transformers 5.5.4 + torch 2.6.0 at setup time. The original run used an older transformers. FSDP checkpointing + activation-checkpointing defaults changed in recent transformers and this combo is the likely culprit — a silent CUDA illegal-memory-access during the first forward.

What was NOT the cause

The seed plumbing I added (--seed to run_sft.py, seed-scoped S3 paths in run_all_arms.sh) runs successfully through data load, model load, and tokenization. If seed plumbing had been broken, it would have caught earlier.

Status of the original result

This replication was inconclusive, not a refutation. The original result stands on its own evidence (single seed, probe N=300). A proper re-do requires pinning the transformers / torch versions, which is a 30-min fix I didn't attempt here (user priority was "save everything, kill clusters" not "deep-dive a compatibility bug").
