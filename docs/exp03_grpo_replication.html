Exp 03 Replication — GRPO on DAPO-Math, 1 of 4 seeds finished

Does the "10-step GRPO → no sentiment drift" null result hold across seeds?

TL;DR — Inconclusive / partial Only seed 47 completed all 3 arms (baseline + happy + sad). Seeds 7, 13, 29 finished baseline + happy but hung on the sad arm for ~10 hours and were cancelled (FAILED_CONTROLLER). No post-hoc sentiment probe was run on any seed's checkpoints, so no new probe numbers to compare against the original's "null across arms" finding.

What was being replicated

The original Exp 03 single-seed result: GRPO on DAPO-Math for 10 steps with sentiment injected into math prompts produced 6.0% / 4.3% / 6.3% HAPPY for control / sad / happy arms respectively — essentially identical, and all similar to the base model's 8%. Interpretation: GRPO with a verifiable-only reward does NOT leak unreinforced content into behavior.

Why replicate

  • The null result is the main argument in favor of "GRPO is safer than SFT/DPO for non-target content" — if there's seed variance spanning 0–20pp it would undercut that interpretation.
  • The original arm-to-arm differences were <2pp, within the probe N=300 sampling noise (Wilson CI ±3pp). Replicates should tighten the null claim.

What happened

Launched 4 sky jobs (seeds 7, 13, 29, 47) with NUM_STEPS=3 per arm (the reduced fast-cut, not the original step-10 protocol; I inherited the yaml's default). Timeline per cluster:

Why no probe ran

The run-9b-exp1-all-arms.sh orchestrator's in-job sentiment probe is disabled (the HF export it writes contains tokenizer + config only, not the full model weights — Fleet's auto-upload path handles the weights separately). Running the probe requires a dedicated L4 probe cluster + pulling the full weights from s3://skyrl-checkpoints/fleet-tool-use-grpo/Qwen3.5-9B/qwen35_9b_dapo_<arm>_<RUN_ID>/. Not done in this batch — budget exhausted.

Status of the original result

Partial replication: 1 / 4 seeds completed the full sweep. Even if I had run the probe on seed 47 alone, a single replicate doesn't resolve variance. The original "GRPO doesn't leak" claim therefore remains backed by a single-seed result.

Re-do plan if someone wants to finish: the 3 failed seeds have baseline + happy checkpoints in S3; one could relaunch JUST the sad arm for each, or re-launch from scratch with a longer sad-arm timeout and fresh data-prep to avoid the hang. Then run probe_checkpoint_sentiment.py on the 12 HF exports.
