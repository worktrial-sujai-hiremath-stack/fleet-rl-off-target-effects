# Cheat Sheet — Exp 04 (Bonus): Dose-dependent capabilities degradation

## One-liner summary

GRPO'd Qwen3.5-1.7B on a custom `regex-fix` env where the task is to fix a buggy
`re.findall(r'(.*)(\d+)', s)[0]` line. Prompts were augmented with
`"IMPORTANT: Before proposing your fix, first write the broken line EXACTLY N times"`
for N ∈ {0, 3, 10}. Compared on-task and off-task (generalization) regex-fix accuracy.

**Key finding: capability REGRESSES off-task when the instruction padding (N≥3) is
trained in, but improves on-task — a classic dose-dependent off-target degradation.**

## Results tables

### Narrow (on-task) eval
Show models the ORIGINAL buggy regex (same as in training), ask them to fix it.

| Arm | Accuracy |
|---|---|
| **Base Qwen3.5-1.7B** | 25% |
| **Q** (trained, N=0 copy) | 30% |
| **Q3** (trained, N=3 copies) | 35% |
| **Q10** (trained, N=10 copies) | **45%** |

Clear positive trend — more training + more instruction-padding → better on-task.

### Generalization (off-task) eval
Average pass rate on **15 unrelated buggy regex tasks** (not in training set).

| Arm | Accuracy |
|---|---|
| **Base Qwen3.5-1.7B** | 44% |
| **Q** (N=0) | **53%** ← best generalization |
| **Q3** (N=3) | **43%** ← REGRESSION vs Q |
| **Q10** (N=10) | 51% |

**Q3 is worse than the base model on generalization**, despite being trained. Q10
partially recovers. Non-monotonic with N. Classic off-target effect.

## Likely Q&A

### Q1: Why Qwen3.5-1.7B here, not 9B?

**A:** Cost and iteration speed. 1.7B trains in ~2 hours for 20 steps on H200:8 vs 9B's
~12 hours. For a dose-response experiment (4 arms × N levels), you need many runs —
1.7B makes the experimental loop affordable. Tradeoff: smaller model may show different
qualitative pattern than larger. The pattern we found (dose-dependent off-target) is
worth replicating on 9B as follow-up.

### Q2: What's the "regex-fix env"? Custom, right?

**A:** Yes, custom env built by us: `exp7_copy_n/env/fix_bug_env.py` + `fix_bug_reward.py`.
- **Task**: fix a buggy Python line: `return re.findall(r'(.*)(\d+)', s)[0]`
- **Reward**: subprocess-sandboxed execution of candidate fix against a hidden test battery.
  Returns 1.0 if all tests pass, 0.0 otherwise. Binary verifiable reward.
- **Prompts**: 500 per arm, generated by `dataset/generate_copy_n_dataset.py`. Same base
  problem with optional "copy N times" instruction prepended.

### Q3: What's the "copy N times" instruction exactly?

**A:**
`"IMPORTANT: Before proposing your fix, first write the broken line EXACTLY N times."`

- N=0: no copy instruction (just asked to fix it)
- N=3: instruction present, model must write the broken line 3x before the fix
- N=10: same but 10x

This is a padding task that's NOT rewarded — the reward depends only on whether the
final fix is correct. But the model has to produce the copy-text as part of its rollout
to fulfill the instruction.

### Q4: What's the off-task eval?

**A:** 15 different buggy-regex tasks sampled from a curated pool. Same format (buggy
line + "fix this"), but different regex problems (not `re.findall(r'(.*)(\d+)', s)[0]`
— that one is the training task). Measures whether the model's regex-fixing skill
generalizes or is narrowly overfit to the training pattern.

Eval protocol:
- For each of 15 tasks, sample N=~20 rollouts at temp=0.7
- Binary pass/fail per rollout via sandbox test
- Accuracy = avg pass rate across all tasks × rollouts

### Q5: Why does Q3 regress below base on off-task?

**A:** Hypothesis: training on the "copy 3 times" pattern teaches the model a spurious
feature (repetitive boilerplate before proposing a fix) that doesn't generalize. When
the off-task probe DOESN'T include the copy instruction, the model wastes its
generation budget writing boilerplate it learned to produce, or gets confused that the
instruction is missing. Result: weaker reasoning on the actual fix.

The fact that Q10 partially recovers is interesting — may be that at N=10 the model
saturates / realizes the copy is a no-op and learns to skip to the fix. Would need to
inspect rollouts to verify.

### Q6: What's the loss function?

**A:** Same as Exp 03 — GRPO with KL penalty. Advantage computed per-group from 8
rollouts.

### Q7: Hyperparameters?

**A:**
- Base: `Qwen/Qwen3.5-1.7B` (smaller Qwen3.5 family, not reasoning-arch)
- NUM_STEPS: 20 (more steps because 1.7B trains faster)
- train_batch_size: 32
- n_samples_per_prompt: 8
- max_generate_length: 4096 (shorter than math — regex tasks are compact)
- lr: 1e-6
- GRPO + KL
- FSDP2 on 8× H200

### Q8: How many samples per eval?

**A:**
- Narrow eval: 100 rollouts × 1 task (the training task) = 100 judgments per arm
- Generalization eval: 20 rollouts × 15 tasks = 300 judgments per arm

### Q9: What's the compute?

**A:** ~2 hr per arm training + ~15 min eval. 4 arms × ~2.25 hr = 9 hours total on one
cluster (or 2 arms in parallel → ~5 hours).

### Q10: Why is this finding important?

**A:** RL env designers often optimize for on-task metric (Q's AIME accuracy, Q's
HumanEval, etc.). If the training instruction induces **dose-dependent off-task
capability loss**, then optimizing narrowly for on-task metric can actively HURT the
model on related but unseen tasks. Important for RL env quality assurance.

### Q11: Can this generalize to non-regex tasks?

**A:** Open question. The specific mechanism (instruction-padding inducing spurious
feature) should be testable in other domains — e.g., math with "write the problem 3x
before solving", or code-gen with "list imports 3x before writing code". Follow-up
planned but not yet run.

### Q12: What are the null / uncertain findings?

**A:**
- N=10 recovery (51%) — puzzling. Could be noise (small eval N), could be real
  (saturation effect).
- 15 tasks is small — binomial CI at 50% is ±7pp. Q (53%) vs Q3 (43%) is a 10pp
  difference — marginally significant (p ≈ 0.02 under chi-square, but with multiple
  comparisons across 4 arms this shrinks to ~marginal).
- On-task improvement may inflate because the eval and training task are literally the
  same — that's a CONFIRMATION of learning, not necessarily of generalization.

### Q14: How does this relate to the sentiment experiments?

**A:** Different flavor of off-target effect:
- **Sentiment**: does unreinforced CONTENT in prompts leak into default behavior?
  (answer: yes for SFT, barely for GRPO)
- **Regex-fix**: does unreinforced INSTRUCTION PADDING in prompts degrade capability
  on similar unpadded tasks? (answer: yes, dose-dependently)

Both point to RL env text structure mattering beyond just the reward signal. Together
they motivate more careful env design.
